## Goals

- Add a self-service password recovery flow: request reset link → receive email → set new password
- Integrate with the existing external email API (POST `/api/v1/send`)
- Reuse better-auth's built-in `sendResetPassword` hook for secure token generation

## Non-Goals

- Rate limiting on the recovery endpoint
- Admin-initiated password resets
- Changes to magic-link or Google OAuth flows

## Approach

Use better-auth's built-in password reset capability by providing a `sendResetPassword` callback in `lib/auth.ts`. This keeps token generation, storage, and validation inside better-auth with no custom DB table needed.

The callback calls the external email API; everything else (client hooks, pages, copy) follows existing patterns (`useSignIn`, `SignInPage`, etc.).

## Key Design Decisions

**Token management via better-auth (not custom):** better-auth already stores reset tokens in the `verification` table when `emailAndPassword.sendResetPassword` is configured. Building a custom table would duplicate this and increase migration risk.

**External email service is server-only:** The email API token (`EMAIL_API_TOKEN`) must never be exposed to the client. The call happens inside the `sendResetPassword` server callback in `lib/auth.ts`.

**Email template name:** The external API is template-driven. We use `"Password Reset"` as the template name and pass `{ resetLink }` as Handlebars content. If the template name differs in production, only `lib/constants.ts` and `lib/auth.ts` need updating.

**Recover password page is accessible without an active OAuth flow:** Unlike `/sign-in`, the `/recover-password` and `/reset-password` pages SHALL NOT require `client_id`/`redirect_uri`. A user who forgets their password before starting a flow must still be able to recover it. After resetting, the user is taken to `/`.

## Flow

```
/sign-in
  └─ "Forgot password?" link → /recover-password

/recover-password
  └─ POST authClient.forgetPassword({ email, redirectTo: "/reset-password" })
       └─ better-auth generates token, calls sendResetPassword(user, url, token)
            └─ server calls POST {EMAIL_API_URL}/api/v1/send
                 { templateName: "Password Reset", to: user.email, content: { resetLink: url } }
  └─ success: show "Check your email" message (no redirect, stateless)

/reset-password?token=<token>
  └─ POST authClient.resetPassword({ newPassword, token })
  └─ success: redirect to /sign-in
```

## Trade-offs

- **better-auth reset tokens are stored in `verification` table** — the table already exists from the sign-in/sign-up setup; no migration required for the table, but better-auth may need a schema extension depending on version. If better-auth doesn't auto-create the token column, a migration will be generated by Drizzle.
- **No confirmation that the email address exists** — `authClient.forgetPassword` returns success regardless (prevents email enumeration). The UI always shows "Check your email."

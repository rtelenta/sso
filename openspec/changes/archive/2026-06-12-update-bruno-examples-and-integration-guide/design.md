## Context

The Bruno collection and integration guide were written for the old custom OAuth implementation. After migrating to `@better-auth/oauth-provider`, all endpoint paths changed and JWT signing changed from HS256 (shared secret) to EdDSA/Ed25519 (asymmetric, JWKS-distributed). Verified from the live OIDC discovery document (`/api/auth/.well-known/openid-configuration`) and JWKS endpoint.

**Live endpoint map (confirmed):**

| Old | New |
|-----|-----|
| `GET /api/oauth/start` | `GET /api/auth/oauth2/authorize` |
| `POST /api/token` | `POST /api/auth/oauth2/token` |
| `POST /api/token/refresh` | `POST /api/auth/oauth2/token` (with `grant_type=refresh_token`) |
| `POST /api/logout` | `POST /api/auth/oauth2/revoke` |

**Token endpoint body format:** `application/x-www-form-urlencoded` only. JSON body is rejected with HTTP 415.

**JWT algorithm:** EdDSA (Ed25519). No shared secret — public key is fetched from `GET /api/auth/jwks`.

**Issuer:** `{SSO_BASE_URL}/api/auth` (e.g. `https://sso.internal/api/auth`). Downstream apps should validate the `iss` claim.

## Goals / Non-Goals

**Goals:**
- All Bruno requests reflect the live plugin endpoints and correct body format
- Integration guide uses accurate endpoint URLs, form-encoded body examples, and EdDSA JWT validation
- Prerequisites table no longer asks for `JWT_SECRET`
- OIDC discovery section added so downstream devs can auto-configure

**Non-Goals:**
- No changes to SSO application code
- No changes to app-side session handling strategy
- No documentation of client_credentials grant (internal server-to-server use, not in scope for this SSO)

## Decisions

### Form-encoded body in Bruno
Bruno supports `body: form-urlencoded`. Switch `02_token-exchange` and `03_token-refresh` from `body: json` to `body: form-urlencoded`. The plugin hard-rejects `application/json` with a clear 415 error — there is no dual-format support.

### JWT validation: EdDSA + JWKS
Old guide told integrators to use a shared `JWT_SECRET` for HS256. New guide will show JWKS-based validation using `jose` (`createRemoteJWKSet`). This eliminates the need to share any secret for JWT verification — integrators only need the `SSO_BASE_URL`.

Recommended pattern for downstream apps:
```js
import { createRemoteJWKSet, jwtVerify } from "jose";

const JWKS = createRemoteJWKSet(
  new URL(`${process.env.SSO_BASE_URL}/api/auth/jwks`)
);

async function verifyAccessToken(token) {
  const { payload } = await jwtVerify(token, JWKS, {
    algorithms: ["EdDSA"],
    issuer: `${process.env.SSO_BASE_URL}/api/auth`,
  });
  return { userId: payload.sub, email: payload.email };
}
```

JWKS responses should be cached locally (the `jose` `createRemoteJWKSet` function handles this automatically with a default 15-minute cache).

### Revocation replaces custom logout
`POST /api/auth/oauth2/revoke` (RFC 7009) is the direct replacement for `POST /api/logout`. Body is form-encoded: `token=<refresh_token>&client_id=...&client_secret=...`. Revocation is idempotent — revoking an already-revoked token returns 200.

The `end-session` endpoint (`POST /api/auth/oauth2/end-session`) is for OIDC RP-Initiated Logout (browser redirect flow) and is not relevant for back-channel revocation.

### OIDC discovery section
Add a short section to the integration guide pointing to `GET /api/auth/.well-known/openid-configuration`. This is the canonical way to get all endpoint URLs and supported algorithms. Downstream apps that use an OIDC client library can auto-configure from it.

### UUID v7 note
The integration guide's JWT claims table already noted `sub` as UUID v7. This remains accurate — the uuidv7 configuration in `lib/auth.ts` ensures all IDs generated by Better Auth (including user IDs) are UUID v7. No change needed for this specific point, but confirm it stays in the guide.

## Risks / Trade-offs

- **JWKS key rotation**: Better Auth rotates JWKS keys periodically. Downstream apps using `createRemoteJWKSet` handle this automatically. Apps that hardcode the public key will break on rotation. The guide should warn against hardcoding the key.
- **`jose` version compatibility**: EdDSA support requires `jose >= 4.x`. The guide already uses `jose` — confirm the example still works.
